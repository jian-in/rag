# RAG 知识库系统 - 架构设计文档

## 整体架构

本系统采用分层架构设计，各层职责清晰、模块可独立替换。

### 数据流

```
                    ┌─────────────────────────────────────────────┐
                    │              知识库构建阶段                    │
                    │                                             │
  文档文件 ──▶ DocumentLoader ──▶ TextSplitter ──▶ VectorStore    │
  (PDF/TXT/     (解析为文本)      (切分为块)       (Embedding +   │
   MD/DOCX)                                      ChromaDB 存储)  │
                    └─────────────────────────────────────────────┘

                    ┌─────────────────────────────────────────────┐
                    │              问答推理阶段                      │
                    │                                             │
  用户问题 ──▶ SmartRetriever ──▶ Reranker ──▶ QAChain ──▶ 答案   │
              (MMR/相似度检索)   (二次精排)    (Prompt +    + 引用  │
                                              LLM 生成)           │
                    └─────────────────────────────────────────────┘
```

### 模块职责

**Ingestion 层**负责将原始文档转化为可检索的文本块。DocumentLoader 使用策略模式根据文件类型选择解析器，TextSplitter 采用递归字符分割策略，优先在段落和句子边界切分，最大限度保留语义完整性。

**VectorStore 层**管理文档的向量化表示和存储。Embedding 模块封装了向量模型的调用，Store 模块管理 ChromaDB 的生命周期（创建、持久化、加载、增量更新）。

**Retrieval 层**实现了两阶段检索架构。第一阶段是粗召回，使用向量相似度或 MMR 算法快速检索候选文档。第二阶段是精排，通过 Reranker 对候选文档进行交叉注意力打分和重排序。

**Chain 层**是 RAG 的核心编排模块。它接收检索结果，与精心设计的 Prompt 模板组合后送入 LLM，生成带有来源引用的答案。

**Evaluation 层**基于 RAGAS 框架提供量化评估能力，支持 Faithfulness、Answer Relevancy、Context Precision 等核心指标的计算。

## 关键设计决策

### 为什么选择 ChromaDB

ChromaDB 是纯 Python 实现的向量数据库，无需额外部署服务。对于作品集项目，它足够轻量且功能完整。在生产环境中可以替换为 Pinecone、Milvus 或 Weaviate。

### 为什么用 MMR 而不是纯相似度检索

纯相似度检索容易返回多个内容高度重复的文档（比如同一内容的不同段落）。MMR 通过引入多样性惩罚，保证返回的文档从不同角度覆盖用户问题，提高 LLM 生成答案的信息量。

### 两阶段检索架构

这是工业界推荐系统的经典架构模式。第一阶段（召回）追求速度，用向量相似度从海量文档中快速筛选出少量候选。第二阶段（精排）追求精度，用更复杂的模型（Cross-Encoder 或 Cohere API）对少量候选进行精细排序。

### Prompt 设计策略

系统 Prompt 明确要求 LLM 必须基于检索上下文回答，不得编造信息，并要求标注来源引用。这有效减少了 LLM 的幻觉问题，同时让用户可以验证答案的可靠性。
